## SAP 테스트 요구 사항 이해하기


사람들은 SAP 테스트라고 하면 흔히 시스템 가동 전 체크리스트를 떠올립니다. [하지만 실제로는 테스트가 훨씬 더 일찍 시작되어 생산 단계까지 계속됩니다.](https://noeldcosta.com/ko/sap-implementation-cost-and-budget-breakdown/)검증 대상은 단순히 소프트웨어만이 아닙니다. 비즈니스가 실제로 어떻게 운영되는지, 즉 거래가 어떻게 이루어지고, 데이터가 어떻게 흐르며, 압박 속에서 사람들이 시스템에 어떻게 의존하는지를 검증하는 것입니다.

### SAP 테스트 유형

모든 SAP 프로젝트에는 다양한 단계의 테스트가 있습니다. 각 단계는 고유한 목적을 가지고 있으며, 어느 한 단계를 무시하면 나중에 예상치 못한 순간에 문제가 발생할 수 있습니다.

*   단위 테스트는 개별 프로그램이나 함수를 대상으로 합니다. 주로 기술적인 테스트이며, 개발자가 코드의 작은 블록이 예상대로 작동하는지 확인하는 데 도움이 됩니다.
*   통합 테스트는 서로 다른 SAP 모듈 간의 통신 방식을 확인합니다. 예를 들어, 판매 주문이 납품 및 청구 단계로 진행될 때 각 단계가 올바르게 연결되어야 합니다.
*   시스템 변경이 발생할 때 회귀 테스트가 중요해집니다. 대부분의 SAP 환경은 서로 연결되어 있기 때문에, [작은 변화가 영향을 미칠 수 있습니다](https://noeldcosta.com/ko/sap-implementation/) 서로 관련 없는 지역들.
*   사용자 수용 테스트(UAT)는 비즈니스 검증에 관한 것입니다. 운영 또는 재무 부서의 사용자들이 시스템이 실제 시나리오를 지원하는지 확인하는 단계입니다.
*   성능 테스트는 많은 팀이 간과하는 부분 중 하나입니다. 시스템은 테스트 중에는 원활하게 작동할 수 있지만, 실제 사용자들이 피크 시간대에 로그인하면 속도가 느려지기 시작합니다. 성능 테스트는 부하, 스트레스, 데이터 볼륨을 시뮬레이션하여 시스템이 항상 응답성을 유지하는지 확인합니다. 이를 통해 어떤 기능 스크립트로도 발견할 수 없는 병목 현상을 찾아낼 수 있습니다.

이것들은 각각 다른 역할을 합니다. [SAP 테스트 세트](https://noeldcosta.com/ko/sap-implementation/s4hana/) 필수 요건입니다. 하나를 다른 하나로 대체할 수는 없습니다. 둘은 독립적으로 작동하는 것이 아니라 함께 작동합니다.

### SAP에서 테스트 자동화의 역할

자동화는 도움이 될 수 있지만 모든 곳에 적용되는 것은 아닙니다. 프로세스가 안정적이고 테스트 케이스가 자주 반복되며 비즈니스에서 명확한 가치를 확인할 수 있을 때 가장 효과적이라고 생각합니다.

자동화를 통해 실질적인 이점을 얻을 수 있는 상황은 다음과 같습니다.

*   잦은 회귀 테스트 주기 실행
*   대규모 데이터 세트 또는 시스템 인터페이스 검증
*   운송 이동 후 예측 가능한 워크플로 재확인

하지만 모든 것을 자동화할 필요는 없습니다. 어떤 테스트는 제대로 작동하는지 판단하기 위해 맥락이나 사용자 경험이 필요합니다. 스크립트는 화면 레이아웃이 사용자를 혼란스럽게 하는지, 워크플로가 실용적인지 여부를 알 수 없습니다.

### 비즈니스 우선순위에 맞춰 테스트를 진행하기

바로 이 부분에서 SAP 검증 프로세스에 더 많은 관심이 필요합니다. 테스트는 단순히 형식적인 절차를 거치는 데 그쳐서는 안 됩니다. 비즈니스에서 실제로 보호해야 할 사항을 반영해야 합니다.

*   월말 결산 기간 동안 중요한 마감일이 있습니까?
*   어떤 프로세스가 매출이나 고객 경험에 직접적인 영향을 미치나요?
*   변경 사항은 전 세계적으로 적용될까요, 아니면 단계적으로 적용될까요?

저는 보통 고객들에게 모든 것을 테스트하는 데 집중하기보다는 가장 중요한 것을 테스트하는 데 집중하라고 조언합니다. 테스트 전략이 비즈니스 목표와 연계되지 않으면 완벽한 테스트 범위조차도 실질적인 보호를 제공하지 못합니다.

SAP 테스트는 제대로 수행될 경우 단순한 기술적 보증을 넘어 운영상의 안정성을 보장하기 때문입니다.

## 주요 SAP 테스트 도구 개요


저는 모든 것이 완벽하게 작동하는 것처럼 보이는 수십 개의 툴 데모를 참관했습니다. 하지만 실제 프로젝트에 들어가면 스크립트가 계속 오류를 일으키거나 통합이 예상대로 작동하지 않는 경우가 종종 발생합니다. 그렇기 때문에 SAP 테스트 툴을 선택할 때는 기능 체크리스트나 벤더에서 제공하는 슬라이드만으로는 판단할 수 없다고 항상 강조합니다.

이 섹션에서는 가장 일반적으로 사용되는 몇 가지 도구를 다룹니다. 모든 경우에 최적의 도구는 아닐 수 있지만, 특히 기능, 지역 및 변경 주기가 다양한 SAP 환경에서 자주 접하게 될 도구들입니다.

### 1\. 트리센티스 토스카

Tosca는 거의 모든 SAP 테스트 도구 비교에서 초반에 언급됩니다. 주로 스크립팅을 사용하지 않고 "모델 기반" 테스트 방식을 사용하기 때문입니다. 이는 팀이 기능 중심적이고 코딩을 원하지 않는 경우에 유용할 수 있습니다. 마치 블록을 조립하듯이 시각적으로 테스트 구성 요소를 구축할 수 있습니다.

나는 그것을 보았다 [풍경 사진에서 정말 뛰어난 성능을 보여줍니다.](https://noeldcosta.com/ko/start-your-sap-implementation-project-right/) SAP는 다른 시스템과 긴밀하게 통합되어 있습니다. SAP GUI, Fiori는 물론 비SAP 애플리케이션까지 단일 테스트 흐름에서 지원하므로 엔드투엔드 테스트에 매우 중요합니다.

GraphQL, gRPC와 같은 최신 프로토콜에 대한 기본 지원이 제한적이며, Postman이나 SoapUI처럼 복잡한 API 체이닝에 필요한 유연성이 부족합니다.

웹 및 UI 자동화

모델 기반 테스트는 안정적인 UI에 적합하며, 시각적 자동화를 통해 다양한 브라우저 환경에서 테스트를 진행할 수 있습니다.

JavaScript를 많이 사용하는 동적 앱(예: React/Angular SPA) 개발에 어려움을 겪으며, 유지 관리 부담이 빠르게 증가할 수 있습니다.

테스트 관리 및 보고

qTest는 테스트 케이스 추적성, 대시보드, Jira 및 CI 도구와의 통합 기능을 제공합니다.

사용자 인터페이스는 시대에 뒤떨어진 느낌을 줄 수 있으며, 기업 보고 기능은 Xray나 Zephyr와 같은 ALM 도구에 비해 기본적인 수준에 그칩니다.

모바일 앱 테스트

Tosca Mobile+를 통한 일부 지원과 Appium과의 통합을 통해 기본적인 워크플로우를 구현할 수 있습니다.

복잡한 동작, 실제 기기 테스트 또는 대규모 크로스 플랫폼 앱 테스트에는 최적화되어 있지 않습니다.

데이터 기반 테스트

강력한 데이터 분리 및 매개변수화 지원 기능을 제공하며, Excel 및 데이터베이스와 원활하게 통합됩니다.

대규모 테스트 데이터 관리(마스킹, 프로비저닝)에는 종종 외부 도구가 필요합니다.

DevOps 통합

Jenkins, Azure DevOps 및 Git 기반 워크플로를 활용하여 테스트 오케스트레이션을 수행합니다.

자동화 파이프라인은 오픈 소스 스택보다 유연성이 떨어지며, 세밀한 제어를 위해서는 스크립팅이 필요합니다.

학습 곡선 및 도입

기술적인 지식이 없는 사용자도 최소한의 코딩으로 테스트를 작성할 수 있으며, 시각적인 흐름 논리를 쉽게 이해할 수 있습니다.

테스트 아키텍처 설정 및 모듈 관리에는 체계적인 거버넌스가 필요하며, 대규모 팀에서는 온보딩에 시간이 걸릴 수 있습니다.

### 2\. Jira용 XRay

XRay는 흥미로운 도구입니다. SAP 전용으로 개발된 것은 아니지만, 팀에서 이미 Jira를 사용하고 있다면 자연스러운 확장 기능처럼 느껴질 수 있습니다. 저는 팀들이 Fiori 앱 개발이나 SAP와 타사 플랫폼 간의 인터페이스 테스트에 XRay를 사용하는 것을 본 적이 있습니다.

이 기능은 Jira 내부에 있으므로 테스트 케이스가 사용자 스토리 또는 변경 요청과 연결됩니다. 이러한 가시성은 특히 애자일 또는 세미 애자일 프로젝트에서 유용합니다.

#### Xray와 Jira가 잘 어울리는 경우(그리고 그렇지 않은 경우)

테스트 영역

엑스레이가 효과적인 곳

엑스레이 촬영의 어려움

Jira를 이용한 테스트 관리

Jira 네이티브 환경에서 원활하게 작업할 수 있으며, 이슈, 요구사항, 결함 및 테스트가 긴밀하게 연결됩니다.

Jira에 지나치게 의존하면 복잡하고 애자일 방식이 아닌 환경이나 외부 테스트 팀과의 협업에서 유연성이 제한됩니다.

수동 테스트 관리

명확한 추적성, 재사용 가능한 테스트 케이스, 버전 관리 및 테스트 세트는 수동 워크플로에서 효과적입니다.

테스트 데이터 기반 반복 작업을 기본적으로 지원하지 않으므로, 수동으로 단계를 중복 수행해야 합니다.

자동화 통합

REST API, Jenkins, Bamboo, GitHub Actions를 통한 CI/CD 지원이 우수하며, JUnit, NUnit, Cucumber와 같은 프레임워크와 통합됩니다.

대규모 테스트 결과 업로드는 관리하기가 복잡할 수 있으며, Tosca나 TestNG Suites와 같은 오케스트레이션 계층이 없습니다.

보고 및 추적성

Jira에는 기본적으로 코드 커버리지 보고서, 요구사항 추적 기능, 테스트 실행 대시보드가 포함되어 있습니다.

복잡한 사용자 지정 보고서를 작성하려면 eazyBI와 같은 외부 도구나 REST API를 사용한 스크립팅이 필요합니다.

BDD/Gherkin 지원

Cucumber 및 Gherkin 구문에 대한 완벽한 지원을 제공하며, 테스트 케이스는 버전 관리되고 사용자 스토리와 연결됩니다.

Jira UI에서 Gherkin 편집 및 가독성은 비즈니스 사용자에게 이상적이지 않으며, 네이티브 IDE 경험이 부족합니다.

확장성 및 성능

애자일 또는 SAFe 환경에서 작업하는 분산된 팀을 지원하는 중간 규모의 Jira 인스턴스에서 뛰어난 확장성을 제공합니다.

수천 개의 테스트 케이스가 있거나 API 상호 작용이 빈번한 대규모 인스턴스에서는 성능이 저하됩니다.

학습 곡선 및 도입

Jira 사용자에게 진입 장벽이 낮고, 이미 Jira를 사용하고 있는 QA 및 제품 팀의 빠른 도입이 가능합니다.

ALM/QC, TestRail과 같은 독립형 ALM 도구에 익숙한 테스트 전문가들은 Jira 기본 모델이 제한적이라고 생각합니다.

### 3\. SAP 솔루션 매니저(SolMan)

이 이야기는 종종 한숨과 함께 나옵니다. SolMan은 오랫동안 존재해 왔고 많은 기업이 설치는 했지만 제대로 활용하지 못하는 경우가 많습니다. 하지만 구조화된 SAP 테스트, 특히 규제가 엄격하거나 통제가 철저한 환경에서는 SolMan이 매우 중요한 역할을 합니다.

SolMan은 요구사항, 테스트 계획, 테스트 실행 및 전송을 모두 한 곳에 연결합니다. 간단해 보이지만 실제로 감사나 시스템 업데이트 중에 발생하는 많은 문제를 해결해 줍니다.

#### SAP 솔루션 관리자가 효과적인 경우(그리고 효과적이지 않은 경우)

테스트 영역

SolMan이 효과적인 곳

솔맨이 어려움을 겪는 곳

SAP 중심 테스트

SAP S/4HANA, ECC 및 BW와의 긴밀한 통합을 제공합니다. 테스트 스위트는 SAP 프로세스 및 전송을 이해합니다.

SAP 생태계 외부에서는 활용 가치가 제한적이며, 다양한 애플리케이션(예: Salesforce, 사용자 정의 포털)과의 교차 테스트에는 적합하지 않습니다.

테스트 자동화(CBTA)

컴포넌트 기반 테스트 자동화(CBTA)는 표준 SAP UI 및 비즈니스 프로세스와 함께 작동하며, 스크립트 없는 실행은 SAP를 인식합니다.

CBTA는 SAP GUI 및 WebDynpro만 지원하며, 최신 브라우저 또는 SAP 이외의 UI는 지원하지 않습니다.

비즈니스 프로세스 변경 분석기(BPCA)

코드 변경 사항을 기반으로 영향을 받는 비즈니스 프로세스를 감지하고, 테스트 범위를 좁혀 테스트 노력량을 줄입니다.

솔루션 문서에 완전하고 최신의 비즈니스 프로세스 문서가 포함되어야 하며, 설정에 많은 리소스가 필요합니다.

테스트 계획 및 실행

중앙 집중식 테스트 계획 실행; 변경 요청 관리(ChaRM) 및 전송과 통합됩니다.

사용자 경험이 시대에 뒤떨어져 있고, 대규모 테스트 주기 관리가 최신 도구에 비해 느리고 복잡합니다.

테스트 문서

요구사항 및 변경 사항과 관련된 수동 테스트 결과, 스크린샷 및 감사 로그를 캡처합니다.

실시간 협업, 버전 관리 또는 테스트 결과물에 대한 다양한 서식 지정 옵션이 부족합니다.

Regression Testing

SAP 분기별 업데이트 또는 전송 번들에 유용하며, Retrofit 및 솔루션 문서와 통합됩니다.

CBTA는 UI 요소의 변화에 대해 상당한 유지 관리가 필요하며, 자체 복구 기능이 없습니다.

도구 도입 및 기술 가용성

SAP ALM 프로세스 교육을 이미 이수한 팀에 가장 적합하며, SAP 운영 환경에 최적화되어 있습니다.

학습 곡선이 가파르다. CBTA와 BPCA는 숙련된 SAP Basis/ALM 팀 외에는 널리 알려져 있지 않다.

### SAP 테스트 도구 사용에 대한 제 추천 사항

저는 일반적으로 SAP 테스트를 문서화와 자동화라는 두 가지 영역으로 구분합니다. 이 두 영역은 서로 다른 문제를 해결해야 하므로 필요한 도구도 다릅니다. 하나의 플랫폼에서 두 가지 모두를 시도하는 것은 제대로 작동하는 경우가 드물며, 그러한 가정 때문에 어려움을 겪는 팀들을 많이 봐왔습니다.

### 테스트 문서화 및 추적성을 위해 SAP 솔루션 관리자를 사용하십시오.

ChaRM 또는 ITSM용 Solution Manager를 이미 사용하고 있다면 테스트 문서화를 위해 기능을 확장하는 것이 좋습니다. SolMan을 사용하면 요구 사항을 테스트 케이스에 매핑하고, 실행을 추적하고, 결과를 전송에 연결할 수 있습니다. 이러한 가시성은 규제 환경이나 변경 사항 추적이 중요한 환경에서 특히 유용합니다.

*   어떤 변경 사항에 대해 어떤 테스트 케이스가 실행될지 제어할 수 있습니다.

*   문서 감사가 더 쉬워지고 유지 관리도 더 간편해집니다.

*   모든 정보는 시스템 변경 로그와 연동되어 한 곳에 저장됩니다.

하지만 이미 더 광범위한 변경 관리 목적으로 SolMan을 사용하고 있지 않다면, 테스트 관리만을 위해 SolMan을 도입하는 것은 권장하지 않습니다. 그런 경우, 도입에 따른 추가 비용이 이점보다 클 가능성이 높습니다.

### SAP 테스트 자동화를 위해 Tricentis Tosca를 사용하세요.

테스트 자동화에는 Tricentis Tosca를 추천합니다. SAP GUI 및 Fiori 워크플로우를 잘 처리하고 엔드투엔드 시나리오 전반에 걸친 회귀 테스트를 지원합니다. 테스트 모델이 안정화되면 실행 일관성이 확보됩니다. 수동 테스트로 인한 오류 발생 위험을 줄이고, 테스트 깊이를 희생하지 않고도 릴리스 주기를 단축할 수 있습니다.

*   토스카는 특히 출시 빈도가 높은 환경에 매우 잘 어울립니다.

*   이 기능은 사용자 지정 스크립트 없이 SAP 시스템과 비SAP 시스템 모두에서 작동합니다.

*   테스트 유지 관리 비용은 기존 자동화 방식에 비해 장기적으로 더 저렴합니다.

제가 보기에 가장 효과적인 방법은 SolMan을 사용하여 문서화 및 추적을 수행하고, Tosca를 사용하여 자동화 및 실행을 수행하는 것입니다.

Worksoft Certify나 Katalon Studio 같은 다른 도구들은 특정 시나리오에서는 여전히 유용할 수 있습니다. Certify는 제약과 같은 검증된 환경에서 강점을 보입니다. Katalon은 간단한 웹 기반 SAP 프로젝트에 도움이 될 수 있습니다. 하지만 이러한 도구들은 핵심 SAP 테스트 프로그램에는 일반적으로 권장하지 않습니다.

## SAP 테스트 도구 선택 시 고려해야 할 요소


SAP 팀들이 전반적인 SAP 테스트 전략과 연계하지 않고 서둘러 도구를 선택하는 경우를 자주 봅니다. 결과는 대개 같습니다. 결국 제대로 활용되지 않거나 워크플로에 마찰을 일으키는 도구를 선택하게 되는 것입니다. SAP 테스트 도구를 선택할 때는 단순히 기능만 비교해서는 안 됩니다. 맥락을 고려해야 합니다.

다음은 제가 고객들에게 도구 세트를 최종 확정하기 전에 검토해 보라고 조언하는 주요 요소들입니다.

### 1\. 조직 목표와의 일치성

제가 가장 먼저 묻는 질문은 "SAP 테스트를 통해 실제로 달성하고자 하는 목표는 무엇입니까?"입니다. 출시 주기를 단축하는 것입니까? 감사 추적 및 규정 준수를 위한 것입니까? 아니면 단순히 글로벌 배포 전반에 걸쳐 회귀 테스트를 더 쉽게 만드는 것입니까?

SAP 테스트 전략을 지원하는 도구는 비즈니스 및 IT 우선순위를 반영해야 합니다. 환경이 자주 변경되거나 여러 단계에 걸쳐 배포되는 경우, 테스트 도구는 반복적인 업데이트를 지원해야 합니다. 반면 안정성이 목표라면 추적성과 구조화된 검증에 더 중점을 두어야 합니다.

이러한 조화가 없으면 아무리 강력한 도구라도 부적합하게 느껴집니다.

### 2\. 기존 시스템 및 워크플로우와의 호환성

테스트 도구가 단독으로는 완벽하게 작동했지만, 팀에서 기존 프로세스에 통합하려고 할 때 오히려 문제를 일으키는 경우를 본 적이 있습니다. 이런 일은 생각보다 자주 발생합니다.

따라서 SAP 테스트 도구를 선택하기 전에 해당 도구가 현재 환경에 어떻게 적합한지 검토해야 합니다.

*   기존 이슈 추적 시스템과 연동되나요?
*   SAP GUI는 물론 Fiori와 같은 최신 인터페이스도 지원할 수 있습니까?
*   기존 사용자들이 별다른 노력 없이도 사용할 수 있을까요?

때때로 해당 도구가 기술적으로는 작동하지만 여러 플러그인을 추가하거나 해결 방법을 사용해야 하는 경우가 있습니다. 이는 장기적으로 비용과 복잡성을 증가시키는 경향이 있습니다.

### 3\. 공급업체 지원 및 커뮤니티 활동

이 부분은 자주 간과됩니다. 저는 도구 선택 기준을 평가할 때 항상 공급업체의 지원 품질을 살펴봅니다. 중요한 배포 중에 문제가 발생했을 때, 팀이 신속하게 실질적인 지원을 받을 수 있을까요?

또한, 해당 도구에 활발한 커뮤니티가 있나요? 동일한 문제를 해결한 사람들이 있나요? 항상 공식적인 지원이 필요한 것은 아닙니다. 포럼 게시글이나 공유된 스크립트가 몇 시간을 절약해 줄 수도 있습니다.

### 4\. 적응성 및 미래 대비

2년 후 여러분의 SAP 환경은 지금과는 완전히 다를 수 있습니다. ECC에서 S/4HANA로 마이그레이션하거나, 클라우드 서비스를 더 많이 도입하거나, 타사 플랫폼을 추가할 수도 있습니다. 따라서 SAP 테스트 도구를 선택할 때는 도구의 적응성을 반드시 확인해야 합니다.

*   SAPUI5와 같은 새로운 UI 프레임워크를 지원합니까?
*   온프레미스 환경과 클라우드 기반 환경 모두에서 사용할 수 있습니까?
*   테스트 전략이 워터폴 방식에서 애자일 방식으로 바뀌더라도 여전히 유용할까요?

저는 고객이 경직된 도구에 얽매이는 것을 피하려고 합니다. 발전 가능성은 나중에 추가하는 것이 아니라 처음부터 설계에 반영되어야 합니다.

궁극적으로 도구 선택 기준은 팀의 업무 방식, 시스템 요구 사항, 그리고 환경 변화의 빈도를 고려하여 결정해야 합니다. 어떤 회사에는 완벽하게 작동하는 도구가 다른 회사에는 오히려 발목을 잡을 수도 있습니다.

### 사례 연구 1: 제조 기업에 Tricentis Tosca 도입하기

SAP ECC와 고도로 맞춤화된 창고 관리 시스템을 운영하는 대규모 글로벌 제조 기업은 테스트를 거친 안정적인 릴리스를 제공하는 데 상당한 지연을 겪고 있었습니다. 긴 회귀 테스트 주기와 높은 결함 유출률로 인해 분기별 릴리스 주기가 자주 중단되었습니다. 대부분의 테스트가 수동으로 진행되어 소수의 핵심 테스터에 대한 의존도가 높아지고 사업부 간 테스트 범위가 일관되지 않았습니다.

#### 도전

*   **테스트 범위가 제한적입니다.** 시간 제약으로 인해 회귀 테스트는 주요 경로에만 집중되었으며, 이로 인해 많은 구성과 예외 흐름이 테스트되지 못했습니다.
*   **수동 작업 의존도:** 모든 테스트 케이스는 수동으로 실행되었으며, 종종 팀원들의 경험적 지식과 버전 관리가 되지 않는 스프레드시트에 의존했습니다.
*   **비즈니스 사용자 참여도 저조:** 비즈니스 관계자들은 이미 과중한 업무에 시달리고 있었고, 테스트 케이스 생성이나 사용자 승인 테스트(UAT) 주기 지원에 시간을 할애하기 어려워했습니다.
*   **일관성 없는 테스트 문서:** 많은 테스트 시나리오에서 명확한 단계나 예상 결과가 부족하여 여러 지역에서 테스트 실행 방식에 차이가 발생했습니다.

#### 구현 접근 방식

*   **파일럿 우선 전략:** Tosca는 타당성을 검증하고 초기 지지를 얻기 위해 위험도가 낮은 기능 영역(입고 물류)에 먼저 도입되었습니다.
*   **모델 기반 테스트 설계:** 테스트 케이스는 재사용 가능한 구성 요소를 사용하여 개발되었으므로 테스터는 이를 다양한 공장 및 사업부에 맞게 신속하게 조정할 수 있습니다.
*   **모듈형 라이브러리 빌드:** 팀은 4주에 걸쳐 SAP 거래 코드 및 주요 비즈니스 프로세스에 맞춰 재사용 가능한 테스트 단계의 중앙 집중식 라이브러리를 구축했습니다.
*   **운송 승인과의 통합:** 자동화된 테스트 실행은 전송 릴리스 워크플로와 연동되어 배포 전에 유효성 검사를 보장했습니다.
*   **단계별 온보딩:** 시범 운영 후, 출고 물류 및 생산 계획과 같은 추가적인 프로세스 영역이 단계적 도입을 통해 테스트 자동화 대상이 되었습니다.

#### 결과

*   **회귀 분석 범위 확대:** 테스트 라이브러리를 통해 대량 거래의 85% 이상을 커버할 수 있었으며, 이는 수동 테스트 방식의 35%에 비해 크게 향상된 수치입니다.
*   **안정적인 릴리스 주기:** 릴리스가 더욱 예측 가능해졌으며, 출시 후 2분기 만에 결함이 40% 감소했습니다.
*   **비즈니스 의존도 감소:** 재사용 가능한 자동화 덕분에 모든 회귀 테스트 주기에서 비즈니스 사용자의 참여 필요성이 크게 줄어들었습니다.
*   **문서 관리 규율 개선:** 자동화된 스크립트는 명확한 문서화를 보장하고 여러 지역에 걸쳐 테스트를 표준화하는 데 도움이 되었습니다.
*   **공장 간 재사용:** 일단 모델이 만들어지면, 최소한의 노력으로 여러 제조 공장에서 쉽게 재사용할 수 있었습니다.

초기 도입 단계에서 약간의 반발이 있었던 것으로 기억합니다. 팀들은 이렇게 특수한 환경에서 자동화하는 것에 대해 회의적이었습니다. 하지만 생산 현장에서 발생하는 사고가 줄어들고, 동일한 테스트 스크립트를 여러 공장에서 얼마나 쉽게 사용할 수 있는지를 확인하자 분위기가 완전히 바뀌었습니다.

### 사례 연구 2: 소매 회사에서 Jira용 Xray 활용하기

한 주요 소매 회사가 최근 [SAP S/4HANA로 마이그레이션됨](https://noeldcosta.com/ko/sap-implementation/s4hana/) 이는 보다 광범위한 디지털 전환 계획의 일환으로 진행되었습니다. ERP 현대화와 더불어, 고객 대면 애플리케이션을 개편하고, Fiori 기반 앱을 도입하며, 다양한 클라우드 서비스와 통합했습니다. IT 전반에 걸친 구현 방식은 애자일 방식이었으며, Jira는 사용자 스토리, 백로그, 테스트 작업 및 제품 릴리스를 위한 단일 기록 시스템 역할을 했습니다.

#### 엑스레이가 효과적이었던 이유

*   **Jira 네이티브 통합:** 테스트 케이스, 실행 및 결함을 사용자 스토리에 직접 연결하여 테스트 범위를 스프린트 계획의 일부로 만들 수 있습니다.
*   **비기술 사용자 접근 권한:** 비즈니스 분석가와 제품 소유자는 도구를 전환하지 않고도 대시보드와 필터를 통해 테스트 진행 상황을 확인할 수 있습니다.
*   **낮은 오버헤드:** 팀들은 테스트 케이스 관리를 위해 별도의 도구를 유지할 필요가 없었습니다. Xray는 기존 Jira 워크플로에 최소한의 마찰만으로 통합되었습니다.
*   **BDD 및 Cucumber 지원:** Fiori 앱을 개발하는 애자일 팀은 Gherkin 구문을 사용하여 승인 기준과 자동화된 테스트를 같은 위치에서 정의했습니다.
*   **팀 간 병렬 테스트:** 테스트 결과물은 가시성이나 추적성을 잃지 않고 팀 간에 재사용하거나 복제할 수 있습니다.

#### 효율성 향상

*   **더 빠른 스프린트 테스트:** 테스트 주기는 스프린트 일정에 포함되어 개발을 저해하지 않으면서 더 빠른 피드백을 가능하게 했습니다.
*   **향상된 가시성:** QA 책임자와 제품 관리자는 실행 진행 상황을 실시간으로 추적하고 위험 요소를 조기에 파악할 수 있습니다.
*   **중복성 감소:** 팀들은 Jira의 내장된 링크 및 레이블 지정 기능을 활용하여 테스트 케이스 중복을 방지할 수 있었습니다.
*   **더 나은 스프린트 리뷰:** 스프린트 데모 중에 테스트 결과를 확인할 수 있었고, 이를 통해 완료 기준에 대한 보다 정보에 입각한 결정을 내릴 수 있었습니다.

솔직히 말해서, 이 솔루션은 배치 작업이나 통합 검증과 같은 대규모 SAP 트랜잭션 테스트에는 적합하지 않습니다. 하지만 Fiori 애플리케이션, API 및 사용자 중심 워크플로의 애자일 테스트에는 추가적인 복잡성을 도입하지 않고도 충분한 성능을 제공했습니다.

### 사례 연구 3: 금융 기관에서 SAP 솔루션 관리자 활용하기

한 대형 금융 서비스 기관은 재무, 자금 관리 및 규제 보고를 포괄하는 고도로 맞춤화된 SAP 환경을 운영하고 있었습니다. 이 기관은 내부 감사자와 외부 규제 기관으로부터 모든 시스템 변경 사항에 대한 완벽한 추적성과 책임성을 확보하라는 압력을 점점 더 많이 받고 있었습니다. 수동 테스트 추적 방식은 특히 테스트를 전송 활동과 연결하거나 감사 시 테스트 증거를 제시할 때 가시성에 공백을 초래했습니다.

#### 초기 문제점

*   **서로 다른 추적 방법:** 테스트 케이스 실행 및 상태는 중앙 저장소 없이 엑셀 시트를 사용하여 관리되었습니다.
*   **자동 감사 추적 기능 없음:** 누가 무엇을 언제 테스트했는지에 대한 증거는 검토 또는 감사 과정에서 수동으로 다시 만들어야 했습니다.
*   **교통편과 연계되어 있지 않습니다.** 테스트가 SAP 전송과 직접 연결되어 있지 않아 릴리스 전에 테스트 결과를 검증하기 어려웠습니다.
*   **개인에 대한 의존도가 높음:** 기관 내 지식이 소수의 시험 담당자에게 집중되어 위험과 관리 부담이 증가했습니다.

#### SAP 솔루션 관리자가 어떻게 도움이 되었나요?

*   **ChaRM에 맞춰 조정된 테스트 계획:** 모든 테스트는 SAP의 변경 요청 관리 프레임워크를 통해 변경 요청 및 변경 문서와 연계되었습니다.
*   **중앙 집중식 실행 추적:** 모든 테스트 활동에는 타임스탬프가 기록되었고, 솔루션 관리자에 저장되었으며, 명확한 소유권 확인을 위해 사용자 자격 증명과 연결되었습니다.
*   **자동화된 문서화:** 내부 감사팀과 규정 준수팀을 위한 보고서는 수동 작업 없이 테스트 계획에서 직접 생성되었습니다.
*   **범위 제어 기능 향상:** 비즈니스 프로세스 변경 분석기를 사용하여 코드 변경의 영향을 평가하고 그에 따른 테스트를 권장했습니다.

#### 비즈니스 영향

*   **감사 준비 상태가 개선되었습니다.** 테스트 범위 보고서와 실행 로그는 감사 중에 요청 시 제공될 수 있으므로 준비 시간을 크게 줄일 수 있습니다.
*   **통제 운송 승인:** 승인자는 변경 사항을 프로덕션 환경으로 이동하기 전에 테스트 상태를 확인할 수 있어 운영 신뢰도가 향상되었습니다.
*   **규정 준수 위험 감소:** 팀 간 테스트 프로세스를 표준화함으로써 테스트가 실제로 수행되었는지, 누가 책임자인지에 대한 의문이 줄어들었습니다.
*   **IT 거버넌스와의 더 나은 연계:** 테스트는 핵심 시스템 외부에서 병렬적으로 처리되는 프로세스가 아니라, 릴리스 라이프사이클의 일부가 되었습니다.

전환점은 감사팀이 더 이상 엑셀 파일을 요구하지 않게 되었을 때였습니다. 감사팀은 솔루션 관리자에서 직접 테스트 이력을 검증할 수 있게 되었고, 이는 향후 IT와 규정 준수 부서의 협업 방식을 완전히 바꿔놓았습니다.

## SAP 테스트 및 검증을 위한 모범 사례


제가 경험한 바로는 SAP 테스트는 구조화되고, 공유되고, 적응 가능할 때 비로소 진정한 가치를 제공합니다. 많은 프로젝트가 도구나 문서화에 노력을 기울이지만, 정작 품질을 향상시키는 데 필수적인 습관을 간과하는 경우가 많습니다. 아래는 제가 실제 프로젝트 현장에서 반복적으로 효과를 확인한 네 가지 SAP 테스트 모범 사례입니다.

### 1\. 명확한 목표와 KPI 정의

방향성 없이 테스트를 진행하면 대개 범위가 좁아지고 기계적인 절차만 거치게 됩니다. 따라서 테스트 케이스를 작성하기 전에 다음과 같은 질문을 던져보는 것이 좋습니다. 무엇을 보호하려는가? 성공적인 테스트 주기란 무엇을 의미하는가?

*   결함 누출률은 테스트를 통과하여 실제 생산 단계로 넘어가는 문제의 수를 보여줍니다. 이 수치가 높으면 테스트 범위 또는 설계에 대한 재검토가 필요할 수 있습니다.
*   [테스트 케이스 재사용 비율](https://noeldcosta.com/ko/sap-project-risk-assessment-matrix-and-mitigation-strategies/) 테스트 라이브러리의 효율성을 파악하는 데 도움이 됩니다. 매번 모든 코드를 다시 작성해야 한다면, 이는 코드 정렬이 잘못되었다는 신호입니다.
*   회귀 테스트 완료 일정은 기한이 정해진 릴리스에서 중요합니다. 전체 회귀 테스트를 완료하는 데 걸리는 시간을 알면 초기에 기대치를 설정하는 데 도움이 됩니다.

이러한 KPI는 단순히 보고용이 아닙니다. 중요한 곳에 에너지를 집중하는 데 도움이 됩니다.

### 2\. 다양한 부서의 이해관계자를 참여시키십시오

SAP 테스트는 한 팀에서만 할 수 있는 일이 아닙니다. 비즈니스 프로세스는 여러 모듈에 걸쳐 있으며, 테스트 계획 또한 그래야 합니다. 비즈니스 팀과 단절된 채 고립된 테스트 전략이 실패하는 경우를 많이 봐왔습니다.

*   기획 단계에서 구매, 재무, 운영 및 IT 부서 담당자들을 참여시키세요. 각 부서 담당자는 서로 다른 관점을 제시합니다.
*   프로세스 담당자를 파악하고 참여를 유도하십시오. 담당자는 자주 발생하는 오류나 예외 상황이 발생하는 지점을 지적해 줄 수 있습니다.
*   비즈니스 사용자를 검토 과정에 참여시키세요. 그들은 구조화된 테스트에서 놓치는 부분을 종종 발견하는데, 단순히 무엇이 잘못되었는지 느끼는 것만으로도 알아챌 수 있습니다.

테스트는 실제 비즈니스 운영 방식을 반영할 때 더 효과적입니다.

### 3\. 테스트에 피드백 루프를 구축하세요

어떤 테스트 계획도 처음부터 완벽할 수는 없습니다. SAP 시스템은 끊임없이 변화하고, 데이터도 변하며, 심지어 사용자 습관까지도 바뀝니다. 효과적인 테스트를 유지하려면 팀은 지속적으로 검토하고 조정할 수 있는 여유를 가져야 합니다.

*   각 주기 후에는 무엇이 실패했는지 살펴보세요. 증상에만 집중하지 말고 근본 원인을 파악하려고 노력하세요.
*   스크립트 내용이 더 이상 유효하지 않거나 관련성이 떨어지면 업데이트하십시오. 스크립트를 고정 자산으로 취급하지 마십시오.
*   팀원들이 테스트 세트 자체에 대해 의문을 제기하도록 하십시오. 일부 단계는 더 이상 실제 사용자 행동을 반영하지 않을 수 있습니다.

피드백 루프가 없다면 테스트는 일상적인 절차가 되어버립니다. 바로 그때 위험이 조용히 스며들기 시작합니다.

### 4\. 테스트 팀을 정기적으로 교육하고 역량을 강화하십시오.

테스트 품질은 결국 테스트를 수행하는 사람에게 달려 있습니다. 아무리 훌륭한 도구를 사용하더라도 테스터가 시스템이나 프로세스를 제대로 이해하지 못하면 중요한 부분을 놓치게 됩니다.

*   테스트 도구 사용 방법에 대한 교육을 제공하세요. 단순히 테스트를 실행하는 방법뿐만 아니라, 테스트를 유지 관리하고 개선하는 방법까지 포함해야 합니다.
*   SAP 비즈니스 프로세스에 대한 이해를 높이세요. 기술 테스터조차도 워크플로의 각 단계가 무엇을 의미하는지 이해해야 합니다.
*   과거 시험 실패 사례와 거기서 얻은 교훈을 공유하세요. 추상적인 교육보다는 이러한 교훈이 더 오래 기억에 남는 경향이 있습니다.

효과적인 SAP 검증은 시간이 지남에 따라 발전하지만, 팀에 대한 지원과 개발이 이루어져야만 가능합니다.

## 맺음말


SAP 테스트 도구를 선택하는 것은 단순히 기능 확인이나 트렌드 추종에 그치는 것이 아닙니다. 귀사의 프로세스, 일정, 그리고 테스트에 대한 기대치에 부합하는 적합한 도구를 찾는 것이 중요합니다.

저는 팀원들이 잠시 멈춰서 진정으로 필요한 것이 무엇인지 정의하는 시간을 가질 때 더 나은 결과를 얻는 것을 보았습니다. 그러한 명확성은 더욱 효과적인 SAP 검증으로 이어집니다. 이는 단순히 도구만으로 얻어지는 것이 아닙니다. 도구를 어떻게 활용하고, 도구가 여러분의 환경을 얼마나 잘 반영하는지에 달려 있습니다.

SAP는 계속해서 변화할 것입니다. 도구들도 생겨나고 사라지기를 반복할 것입니다. 진정한 이점은 유연성을 유지하고, 작은 조정을 자주 하며, 비즈니스 목표에 기반한 테스트 접근 방식을 유지하는 데 있습니다.

SAP 테스트 도구 선정 과정을 직접 경험하셨거나 구현 과정에서 어려움을 겪으셨다면, 그 경험을 공유해 주시면 감사하겠습니다. 편하게 연락 주시거나 댓글을 남겨주시면 좋겠습니다.

결국, 그러한 삶의 경험은 안내서에 적힌 어떤 내용보다 더 많은 것을 가르쳐주는 경우가 많습니다.

### 1\. 자동화를 위한 최고의 SAP 테스트 도구는 무엇입니까?

내 경험으로 볼 때, **트리센티스 토스카** SAP 테스트 자동화에 가장 효과적인 도구로 손꼽힙니다. 모델 기반 접근 방식을 통해 특히 복잡한 SAP 환경에서 테스트 케이스 생성 및 유지 관리를 간소화할 수 있습니다. Tosca는 다양한 SAP 모듈에 걸쳐 엔드투엔드 테스트를 처리할 수 있는 기능을 갖추고 있어 강력한 자동화 솔루션을 찾는 조직에게 믿을 수 있는 선택입니다.

### 2\. SAP 솔루션 매니저는 테스트 자동화를 지원할 수 있습니까?

DaVinci에는 **SAP 솔루션 매니저(SolMan)** 일부 자동화 기능을 제공하지만, 주로 테스트 관리 및 문서화를 위해 설계되었습니다. 포괄적인 테스트 자동화를 위해서는 SolMan을 Tricentis Tosca와 같은 도구와 통합하면 더욱 확장 가능하고 효율적인 솔루션을 제공할 수 있습니다.

### 3\. Tricentis Tosca는 SAP와 어떻게 통합됩니까?

Tricentis Tosca는 SAP GUI, Fiori 및 기타 SAP 기술을 지원하여 SAP와 완벽하게 통합됩니다. 이 기능을 통해 SAP 환경 내에서 직접 실행할 수 있는 자동화된 테스트 케이스를 생성할 수 있으므로 정확하고 효율적인 테스트 프로세스를 보장합니다.

### 4\. Tricentis Tosca와 SAP Solution Manager를 모두 사용해야 합니까?

두 가지 도구를 모두 사용하는 것이 유익할 수 있습니다. **SAP 솔루션 관리자** 테스트 문서, 요구사항 및 변경 관리 분야에서 탁월한 능력을 발휘하며, **트리센티스 토스카** 강력한 자동화 기능을 제공합니다. 이 둘을 통해 관리 및 실행 측면을 모두 포괄하는 종합적인 테스트 프레임워크를 제공합니다.

### 5\. Tricentis Tosca를 SAP 테스트에 사용할 때의 주요 이점은 무엇입니까?

Tricentis Tosca는 다음과 같은 여러 장점을 제공합니다.

*   **모델 기반 테스트**: 테스트 케이스 생성 및 관리를 간소화합니다.

*   **위험 기반 테스트**: 검사 노력을 최적화하기 위해 위험도가 높은 지역에 집중합니다.

*   **통합 기능**: 다양한 SAP 모듈 및 기타 기업용 애플리케이션과 원활하게 통합됩니다.

이러한 기능은 SAP 테스트 프로세스를 더욱 효율적이고 효과적으로 만드는 데 기여합니다.

### 6\. SAP Solution Manager는 테스트 문서화를 어떻게 지원합니까?

SAP Solution Manager는 테스트 문서 관리를 위한 중앙 집중식 플랫폼을 제공합니다. 이 시스템은 테스트 케이스의 생성, 저장 및 추적을 가능하게 하여 추적성과 규정 준수를 보장합니다. 이러한 중앙 집중식 접근 방식은 팀 간의 협업을 향상시키고 보다 체계적인 테스트 프로세스를 가능하게 합니다.

### 7\. Tricentis Tosca는 SAP 이외의 애플리케이션에도 사용할 수 있습니까?

네, Tricentis Tosca는 다목적 솔루션으로 SAP 외에도 웹, 모바일, 데스크톱 애플리케이션 등 다양한 애플리케이션 테스트를 지원합니다. 이러한 특징 덕분에 다양한 기술 환경을 가진 조직에게 유용한 도구가 됩니다.

### 8\. Tricentis Tosca를 구현할 때 어떤 어려움에 직면할 수 있을까요?

몇 가지 일반적인 과제는 다음과 같습니다.

*   **학습 곡선**: 팀이 해당 도구를 능숙하게 사용하는 데 시간이 걸릴 수 있습니다.

*   **초기 설정**: 기존 프로세스에 맞춰 Tosca를 구성하는 것은 복잡할 수 있습니다.

*   **통합**: 다른 도구 및 시스템과의 원활한 통합을 보장하려면 추가적인 노력이 필요할 수 있습니다.

이러한 과제를 해결하려면 실행 단계에서 적절한 계획, 교육 및 지원이 필요합니다.

### 9\. 테스트 자동화는 SAP 프로젝트 성과를 어떻게 향상시키나요?

테스트 자동화는 다음과 같은 방식으로 SAP 프로젝트 성과를 향상시킵니다.

*   **수작업 부담 감소**: 자동화된 테스트는 시간과 자원을 절약해 줍니다.

*   **테스트 범위 확대**: 더 많은 시나리오를 효율적으로 테스트할 수 있습니다.

*   **정확성 향상**: 자동화는 테스트 과정에서 발생하는 인적 오류를 최소화합니다.

*   **배송 속도 향상**: 테스트 주기가 빨라지면 프로젝트 완료 속도도 빨라집니다.

이러한 개선 사항은 더 높은 품질의 결과물과 더 성공적인 SAP 프로젝트로 이어집니다.
